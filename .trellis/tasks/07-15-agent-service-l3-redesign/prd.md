# Redesign Agent Service L3 architecture

## Goal

Research and design an Agent Service/L3 refactor whose core migration remains internally compatible, while also planning the explicitly approved, separately releasable Portal/API slice that lets a user cancel the current Agent Turn. The design improves cross-domain extensibility and accommodates memory, skills, MCP, and future agent capabilities.

## Requirements

DL-01 governs all R-001 through R-176 implementation work. Before implementation begins, every phase must be decomposed into one-to-three-engineer-day slices with an owner, explicit dependencies, rollback boundary, and pre-committed binary acceptance check; each slice records estimated and actual duration.

- **R-001 — Preserve the state-authority hierarchy.** Product workflow facts remain authoritative in the Agent Service workflow model; physical execution facts remain authoritative in Nexus/Lab; Session Events remain an append-only audit, UI projection, and replay record; Agent execution state remains non-authoritative working state.
- **R-002 — Keep progress entity-authoritative and event-projected.** The refactor must not promote Session Events or LangGraph checkpoints into a competing source of workflow truth.
- **R-003 — Preserve projection semantics.** Portal live events, cold-loaded snapshots, and replayed progress must remain semantically equivalent even when their transport representations differ. Transient stream deltas are presentation-only and have no replay requirement; every durable event kind must declare its commit class and projection path.
- **R-004 — Preserve L1-L4 responsibilities, not topology.** The refactor must retain the responsibility and dependency direction of L1 protocol adaptation, L2 cross-turn coordination and deterministic workflow change, L3 turn-scoped Agent execution, and L4 technical adaptation. Existing file placement, graph topology, and Chemistry specialist wiring are not compatibility constraints.
- **R-005 — Make architecture boundaries executable.** L1-L4 dependency and ownership rules must be enforced by automated architecture checks in CI rather than documentation and review alone.
- **R-006 — Keep one layer taxonomy.** L1-L4 remain the only architectural layers. Agent Foundation, Domain Pack, and Proposal Boundary are orthogonal execution, extension, and effect-governance boundaries rather than a second layer stack.
- **R-007 — Place Agent Foundation inside L3.** Agent Foundation is the domain-neutral kernel for turn-scoped Agent execution and exposes ports needed by Domain Agent Definitions; it does not own deterministic workflow transition policy.
- **R-008 — Isolate domain implementations in Domain Packs.** Chemistry, Biology, and later experimental domains provide their versioned schemas and behavior through Domain Pack contracts instead of being encoded in Agent Foundation.
- **R-009 — Use one Proposal Boundary between L3 and L2.** L3 may describe a desired side effect only as a typed Proposal. L2 reloads authoritative facts and owns authorization, concurrency checks, deterministic transition, transactional workflow-event append, and outbox creation. Durable conversation or observation output uses the L2 Turn-output path and is not a Proposal.
- **R-010 — Do not create a parallel mutation pipeline.** Proposal handling must converge on the existing L2 deterministic transaction and reducer path during migration; it must not introduce a second workflow authority, event log, or state-change pipeline.
- **R-011 — Implement optional platform capabilities only through demand gates.** Long-term Memory, a general Skill runtime, general MCP enablement, and durable Agent execution are not initial-refactor deliverables. Each requires a named consumer, a concrete use case, measurable benefit, and authority/safety acceptance criteria before implementation.
- **R-012 — Keep initial Foundation scope evidence-based.** The first Agent Foundation version contains only capabilities exercised by the existing Chemistry workflow or the agreed cross-domain validation slice; it must not include unused Store, registry, gateway, or durable-thread implementations.
- **R-013 — Keep outward Foundation contracts framework-neutral.** BIC-owned semantic contracts govern Foundation seams to L2, Proposal handling, and Domain Packs. Framework types, including `langchain*`, `langgraph*`, and `pydantic_ai*`, must not cross those seams. Foundation adapters may use supported public framework interfaces internally when they satisfy the same BIC provider contract.
- **R-014 — Require a second-domain validation slice.** Agent Foundation public SPI, Experiment Workflow Template/Kit contracts, and Domain Pack v1 contracts must not be declared stable until a minimal Biology slice exercises the complete internal lifecycle from objective through report without embedding Biology behavior in Agent Foundation or the Experiment Workflow Kit.
- **R-015 — Keep the Biology slice inside the refactor boundary.** The validation slice uses Agent Service contract harnesses and a Lab simulator behind production-shaped ports. It does not authorize changes to Portal, Lab Service, BIC-shared-types, or a production Biology product.
- **R-016 — Shape the provisional contracts with Chemistry, then challenge them with Biology.** The migration first runs a complete Chemistry vertical slice through the new architecture and incrementally moves the remaining Chemistry paths while the Agent Foundation public SPI, Experiment Workflow Template/Kit contracts, and Domain Pack contract remain provisional. Biology exploration has not started and supplies no real examples, so inventing the shared skeleton from hypothetical Biology behavior would create more rework than challenging a Chemistry-shaped provisional contract later. A minimal Biology slice must exercise the same architecture before those v1 contracts freeze; all migrated Chemistry paths must pass again before the one-time default switch and legacy deletion.
- **R-017 — Require zero coordinated external changes for the core Chemistry refactor.** Each core migration stage must remain deployable against the selected compatibility baseline without required changes to Portal, Lab Service, or BIC-shared-types. The separately scoped user-cancellation product slice in R-093 through R-096 is the only currently approved exception.
- **R-018 — Preserve semantic external compatibility with one ruled failure-path exception.** Compatibility covers current Chemistry REST, snapshots, SSE kinds/payloads/order/replay/errors, Lab MQ/HTTP/authentication/identity/idempotency, shared schemas, evidence propagation, locale, confirmation gates, and ELN gating. Byte-identical internals and unobservable timing are not frozen. ADR-0012 intentionally replaces a current partially committed workflow-action failure prefix with all-or-nothing Proposal semantics; each affected action must record the old and target failure trajectories in its compatibility baseline.
- **R-019 — Review PR-sourced wire changes independently.** Every externally visible field, event, error, or ordering change from PR #94, PR #136, or later candidate work receives an explicit adopt/adapt/drop decision. No such change is inherited merely because its internal mechanism is adopted.
- **R-020 — Separate current-product refactoring from future-domain product expansion.** Production Biology support and any required cross-repo contract expansion belong to a separately approved roadmap; they are not implied by the internal Biology validation slice.
- **R-021 — Give the outbox executor exclusive external-command authority.** L3, Domain Packs, tools, Skills, Proposal handlers, and workflow transactions must not call Lab/Nexus or another state-changing external command endpoint directly. Only the outbox executor may perform an External Command.
- **R-022 — Persist command intent atomically.** An accepted effectful Proposal must commit its deterministic workflow transition, ordered workflow-transition Session Events, and durable Outbox Command in one L2-owned transaction before any external call occurs.
- **R-023 — Gate retries by command type.** An Outbox Command may be executed once without automatic retry. Automatic retry for a command type remains disabled until the receiving system is verified to accept and persist the same BIC idempotency key.
- **R-024 — Correlate before dispatch and acknowledge after receipt.** External commands use a precommitted BIC command identifier for callback correlation. Existing dispatched semantics are emitted only after the external system returns its stable task identifier and the corresponding Command Receipt is committed.
- **R-025 — Treat PR #94 and PR #136 as candidate parts and supersede PR #150.** The redesign must review #94 and #136 through an explicit adopt/adapt/drop matrix; neither is rebased or merged wholesale as the architecture baseline. This cross-repo design supersedes the closed Agent Service PR #150, whose branch remains historical evidence of the three-round Claude × Codex review; only its ruled Framework Ruling and rolling-summary commitment carry forward explicitly.
- **R-026 — Converge on one Proposal taxonomy.** Action, effect, outcome, and reason concepts adopted from the two PRs must map to one versioned Proposal model and cannot survive as parallel policy or engine vocabularies.
- **R-027 — Keep admission non-authoritative and domain policy domain-owned.** Useful #94 policy outcomes, reason codes, and validated rules are adapted into the Chemistry Domain Proposal Policy. Admission and middleware checks may provide early UX feedback but cannot authorize a workflow change.
- **R-028 — Place #136 mechanisms behind the agreed boundaries.** CAS/locking, derived views, engine logic, and outbox infrastructure may be adapted into the L2 Proposal transaction and Chemistry Domain Pack. Chemistry-hard-coded Baton, transition, registry, synchronous Lab dispatch, and unapproved wire additions are not inherited as Foundation contracts.
- **R-029 — Bound Agent Foundation v1 to execution mechanics.** Foundation v1 may provide model integration and centrally governed resolution of the graph-declared `light` or `complex` Model Capability Level, immutable Invocation Context, per-Turn working-context reconstruction, lean Agent State, framework-neutral execution and stream contracts backed by the current LangGraph adapter, context-block assembly and budgeting, stored rolling-summary consumption, execution telemetry and Model Invocation Provenance, deadlines, cancellation, one typed `Foundation Execution Outcome` for each invocation, and a minimal standard-tool wrapper for principal propagation, telemetry, and side-effect classification. The Turn Working Context is rebuilt under the current claim from the immutable Invocation Context, protected typed fact projections current at reconstruction time, and admitted projections/contributions; it is neither a mutable Session object nor commit authority. A Foundation Execution Outcome reports L3 execution closure to L2; it is not the durable L2 `Turn Terminal Outcome`.
- **R-030 — Exclude workflow and domain ownership from Foundation.** Foundation v1 must not own workflow persistence, CAS, outbox, external commands, domain schemas/prompts/forms/workflows/reports, or concrete Mind/Lab/MinIO clients.
- **R-031 — Exclude unapproved future-capability implementations from Foundation v1.** Foundation v1 must not contain a Memory Store, general Skill registry, general MCP gateway, or durable checkpointer implementation.
- **R-032 — Confine framework integration to adapters.** LangGraph remains the non-authoritative, operationally load-bearing runtime through Phase 1. Its adapter must use supported public interfaces and remove `langgraph._internal` and undocumented lifecycle/namespace dependencies. Bidirectional tests must reject framework types at outward seams and prove each permitted adapter satisfies the BIC-owned contract.
- **R-033 — Package Domain Packs as trusted static Python packages.** Domain Pack v1 components are installed and registered at application composition time; runtime hot loading and untrusted third-party plugin execution are out of scope.
- **R-034 — Require a versioned Domain Pack Manifest.** Each pack declares its domain identity/version, compatible Agent Foundation public SPI, Experiment Workflow Template/Kit, and Proposal contract ranges, its Domain Agent Definition and Domain Proposal Policy factories, typed Domain Step capabilities, shared neutral contract types, permissions, and external ports. Every hosted graph declaration selects exactly one of `light` or `complex`; the Manifest cannot declare an arbitrary model-capability bag, provider, model identifier, endpoint, or deployment.
- **R-035 — Implement domain behavior through typed ports and factories.** Schemas, Agents, typed Domain Step capabilities, policy, tools, context providers, evidence adapters, and deterministic report contributors remain ordinary typed Python implementations behind Agent Foundation, Experiment Workflow Kit, and Proposal contracts.
- **R-036 — Do not invent a general workflow DSL in v1.** The manifest describes and validates composition; it does not replace domain behavior with a new YAML/JSON execution language.
- **R-037 — Keep concrete domains out of neutral-core composition.** Adding or enabling a Domain Pack may change trusted deployment registration but must not require a concrete-domain import, enum member, conditional, graph edge, or domain-specific schema branch in Agent Foundation or generic L2 orchestration/adjudication.
- **R-038 — Limit model authority to domain intent.** An LLM, Agent, Skill, or model-visible tool may select a Proposal kind and supply only the domain payload permitted by its typed schema. It cannot assert its own identity, capabilities, authoritative target, concurrency precondition, Proposal identity, or idempotency identity.
- **R-039 — Construct Proposal Envelopes in trusted runtime code.** A trusted Proposal Factory binds the typed intent payload to actual Principal Context, workflow/session/turn/run references, domain and component versions, expected-version/CAS preconditions, provenance, Proposal identity, and idempotency identity.
- **R-040 — Re-authorize every Proposal against current facts.** Fields produced by the trusted Proposal Factory are inputs to, not substitutes for, the L2 authoritative decision. L2 must reload or lock the required current-fact inputs—including current Workflow Facts and any catalogued correlated fact projections required by that action—and validate the envelope, capabilities, domain policy, and preconditions before acceptance.
- **R-041 — Classify every model-facing tool before exposure.** Tools use exactly one execution category: Pure Tool, Read-Only Query Tool, Proposal Tool, or External Command. Pure and read-only tools may execute during an Agent invocation; Proposal Tools produce Intent Payloads; External Commands are never model-visible.
- **R-042 — Govern read-only queries through Principal Context.** A Read-Only Query Tool must be explicitly registered and reviewed as read-only, declare required capabilities and authoritative source, receive credentials/tenant/actor from trusted runtime context, and enforce schema validation, allowlisting, deadlines, rate limits, auditing, and telemetry. An optional Query Agent, when configured, is a hosted `light` graph whose graph-specific tool grants are limited to Pure Tools and Read-Only Query Tools. It receives neither a Proposal Tool nor the Proposal Port, External Command adapter, Persistence handle, mixed read/write client, mutation credential, or workflow-policy capability, and intent routing cannot widen those grants.
- **R-043 — Preserve query-result provenance and freshness.** Read-only results supplied to an Agent must identify their authority, source reference, observation time, and freshness/staleness information and must fail loudly when the authoritative query fails.
- **R-044 — Do not promote query snapshots into Workflow Facts implicitly.** A query result is a non-authoritative observation in Agent context. Persisting or acting on it as a workflow transition requires a separate Proposal adjudicated against current facts.
- **R-045 — Give each logical Proposal a stable trusted identity.** The Proposal Factory creates `proposal_id` once, persists it in lean Agent State before submission, and preserves it across execution attempts for the same logical tool call. Identity binds the stable Agent run, tool call, Proposal kind, and target workflow rather than relying on payload hash alone.
- **R-046 — Derive downstream idempotency from Proposal identity.** An accepted transition records internal Proposal provenance, and each resulting Outbox Command derives a stable identity from `proposal_id`, command kind, and command ordinal. Database uniqueness prevents a duplicate Proposal from creating a second logical transition or command.
- **R-047 — Keep Proposal receipts minimal in v1.** V1 does not introduce a generic Proposal Receipt aggregate without a demonstrated query or recovery need. Accepted Proposals use persisted transition/outbox provenance as their durable receipt; non-accepted adjudications are typed and audited without modifying Workflow Facts.
- **R-048 — Derive the final Proposal Outcome taxonomy from a complete action-state catalog.** Exact outcome types and names are not yet frozen. Before implementation, the design must enumerate every workflow action, eligible current state, precondition, authoritative owner, typed outcome, aggregate mutation, Session Event, Outbox Command, external response, idempotency behavior, and replay behavior.
- **R-049 — Define a Turn as one bounded normalized-input execution.** A Turn has an identity when enqueued, begins execution when L2 dequeues that input, and ends when L2 durably records its unique `Turn Terminal Outcome`. Queue wait is observable separately from execution duration.
- **R-050 — Keep Agent work inside the Turn boundary.** Context loading, LLM reasoning, Pure and Read-Only Query Tool execution, Proposal construction, and synchronous L2 Proposal adjudication caused by that input may occur within the Turn. Governed Query Result Snapshots and lean Agent State may extend only the current Turn Working Context. An execution-eligible reclaim reconstructs that context under the new claim generation, while an elapsed-deadline reclaim skips Foundation and competes for timeout closure; a later input creates a new Turn context. None of these paths resumes hidden mutable Session state.
- **R-051 — Keep asynchronous external execution outside the Turn boundary.** Outbox execution and Lab/Nexus Task lifecycles continue independently after the originating Turn can end. Their callback or reconciliation input creates a new Turn with a new `turn_id`.
- **R-052 — Use distinct lifecycle identities.** Turn, Proposal, Outbox Command, and Lab/Nexus Task identities and statuses must remain separate; no identifier or status field may implicitly represent more than one lifecycle.
- **R-053 — Use in-process strong contract isolation for Foundation v1.** Agent Foundation and trusted static Domain Packs may run in the Agent Service process, but process co-location must not grant access to workflow persistence, concrete L2 services, outbox repositories, or state-changing external clients.
- **R-054 — Inject only governed execution ports into L3.** Foundation and a Domain Pack's L3-facing Domain Agent Definition may receive Pure Tool capabilities, governed Read-Only Query ports, and a typed Proposal adjudication port. They must not receive raw Lab/Nexus clients, repositories, transactions, or service containers from which an effect bypass can be recovered. The L2-facing Domain Proposal Policy receives only neutral typed Proposal/current-fact views and cannot receive L3 execution capabilities or infrastructure.
- **R-055 — Keep Proposal adjudication transactionally isolated from model execution.** L2 may synchronously adjudicate a Proposal during a Turn and return a typed outcome, but no database transaction, entity lock, or outbox lease may remain open while awaiting an LLM, narration, graph continuation, or read-only external query.
- **R-056 — Do not propagate Turn cancellation across a committed effect boundary.** Cancellation before Proposal commit may prevent submission. Once a Proposal transaction has committed an Outbox Command, the command retains its own lifecycle; cancelling the originating Turn cannot erase, roll back, or implicitly cancel it.
- **R-057 — Enforce isolation through multiple mandatory CI gates.** Static lint, import-linter dependency contracts, type/architecture tests, Composition Root tests, and startup manifest validation must jointly enforce the boundary. Documentation or code review alone is insufficient.
- **R-058 — Default-deny unclassified capabilities.** An import dependency, injected runtime capability, model-visible tool, Domain Pack component, or MCP tool that is not explicitly admitted by the applicable contract must fail CI or startup validation rather than inherit access through a generic client or container.
- **R-059 — Demand-gate process or sandbox isolation.** A separate Foundation process or sandbox is not a v1 prerequisite. It becomes a design requirement only when an approved use case introduces untrusted executable Skill/MCP/plugin code or another threat model that in-process contracts cannot satisfy.
- **R-060 — Permit at most one accepted effectful Proposal per Turn.** L2 must durably prevent concurrent, repeated, or replayed Proposal calls from committing more than one accepted business action for the same `turn_id`.
- **R-061 — Do not count rejected candidates as accepted effects.** An Agent may receive a typed rejection and submit a corrected candidate while the Turn has no accepted Proposal. Retrying delivery of the same logical candidate preserves Proposal identity and reconstructs its prior outcome; a materially corrected intent is a new candidate identity.
- **R-062 — Close effect capability after acceptance.** Once an accepted Proposal transaction commits, later Proposal calls in that Turn cannot mutate state or create commands. The remaining invocation is restricted to pure/read-only behavior and grounded narration before execution closure.
- **R-063 — Model one Proposal as one business action, not one row write.** The accepted Proposal may atomically update multiple Workflow Facts, append the ordered externally compatible Session Events required by that action, and create one or more correlated Outbox Commands defined by the reviewed action-state catalog.
- **R-064 — Record current multi-effect behavior as migration debt.** The migration plan must not assume the existing runtime already provides one-effect semantics. It must inventory per-event transactions, direct L3 persistence, parallel submit calls, deterministic TLC commands, and any Turn path that can sequence multiple mutations or commands.
- **R-065 — Separate durable Input Admission from Turn execution.** L2 must durably accept a normalized user/system/callback input before asynchronous Agent execution begins.
- **R-066 — Commit input facts, receipt, and queued Turn atomically.** One admission transaction must perform applicable validation/deduplication/CAS, apply the deterministic user or external-fact transition, append its input receipt or Session Event, and create the immutable identified Turn work item in queued state.
- **R-067 — Acknowledge only committed admission.** HTTP success or MQ acknowledgement must not occur until the admission transaction commits. A validation, authorization, ownership, deduplication, or CAS rejection that does not admit work creates no Turn.
- **R-068 — Recover admitted Turns independently of process memory.** A committed queued Turn must be claimable after process restart and must eventually reach one durable `Turn Terminal Outcome` even if the original process fails before dequeue.
- **R-069 — Keep admission effects outside the L3 Proposal quota.** A deterministic fact change caused directly by the admitted input is owned by the admission transaction and does not consume the Turn's single accepted L3 Proposal slot. The two authorities and their provenance remain explicit.
- **R-070 — Separate queue and execution timing.** `turn_id` is allocated during admission; admission/queue latency is observable independently, and the Turn execution deadline begins only when a worker successfully claims the queued Turn.
- **R-071 — Fence every durable Turn claim.** The durable Turn row carries an internal monotonically increasing `claim_generation` plus lease owner/expiry. Every initial claim and reclaim atomically increments the generation.
- **R-072 — Remove stale-claim authority everywhere.** Proposal adjudication, effect-slot closure, Agent-output persistence, `Turn Terminal Outcome` CAS, and user-visible stream emission must verify the current `(turn_id, claim_generation)`. A stale claimant cannot mutate, close, or narrate the Turn.
- **R-073 — Keep claim fencing operational, not a new business lifecycle.** V1 must not add a first-class Attempt aggregate/table, Attempt Session Events, public attempt identity, or durable Agent checkpoint solely to implement recovery fencing.
- **R-074 — Demand-gate persisted claim history.** Per-claim logs/traces are sufficient until a named audit, billing, debugging, or mid-execution recovery consumer requires durable historical claim records.
- **R-075 — Keep the Turn operational state machine minimal.** The durable Turn row uses only `queued`, `running`, and `terminal` operational states.
- **R-076 — Express recovery through claim metadata, not lifecycle proliferation.** Initial claim moves `queued -> running`; reclaim of an expired lease retains `running` while replacing owner/lease and incrementing `claim_generation`. V1 must not add retrying, orphaned, recovering, or similar Turn states.
- **R-077 — Separate terminal category from operational state.** Completion, failure, timeout, and cancellation are represented by the typed `Turn Terminal Outcome` attached to `terminal`, not as additional row states.
- **R-078 — Close the Turn row and terminal event atomically.** L2 must use one fenced CAS transaction to set the unique `Turn Terminal Outcome` and append the externally compatible terminal Session Event. A losing writer reads the existing outcome and cannot append another terminal event.
- **R-079 — Keep queue and reclaim lifecycle internal.** Queue admission, lease renewal, and reclaim produce metrics/traces but require no new Portal-visible event or shared-type change in the Chemistry refactor.
- **R-080 — Keep lifecycle outcomes orthogonal.** Turn Terminal Outcome, Proposal adjudication, Outbox Command execution, and Lab/Nexus Task execution must retain independent typed outcomes. No aggregate status may overwrite or impersonate another lifecycle.
- **R-081 — Preserve committed effects after Turn failure.** If a Proposal commits before later Agent narration, streaming, or cleanup fails or times out, its Workflow Fact changes, events, effect-slot closure, and Outbox Commands remain valid and recoverable.
- **R-082 — Do not infer business failure from Agent closure failure.** A failed/timed-out Turn after Proposal commit describes incomplete Agent execution or response. Portal projections and operators must still use the Proposal, Command, and Task records to determine business/physical progress.
- **R-083 — Correlate rather than duplicate outcome state.** The Turn row may link to the accepted Proposal through the effect slot/provenance, but `Turn Terminal Outcome` fields must not copy Proposal, Command, or Task status into a composite total-state enum.
- **R-084 — Treat Turn Terminal Outcome fields as provisional until consumer review.** Candidate outcome kinds, reason fields, response-completeness fields, stage/component fields, and diagnostic references are design hypotheses rather than contracts until the Turn Terminal Outcome case catalog and named-consumer matrix are approved.
- **R-085 — Do not persist topology labels without a consumer.** `failure_stage` or an equivalent graph/component location remains telemetry-only unless a named recovery, Portal, audit, or operations decision requires a stable persisted distinction that semantic reason and trace data cannot provide.
- **R-086 — Derive the Turn Terminal Outcome schema from exhaustive cases.** The design must review normal completion, rejection, pre/post-effect failure, deadline, cancellation, reclaim, stale claim, and terminal-persistence failure before freezing `Turn Terminal Outcome` types or fields.
- **R-087 — Require a Field Necessity Record.** Every proposed durable or externally visible `Turn Terminal Outcome` field must answer the five mandatory questions about named consumer, decision/action, insufficiency of existing facts/trace, compatibility exposure, and cross-domain/topology stability before acceptance.
- **R-088 — Fix one absolute execution deadline per logical Turn.** The first successful claim sets `execution_deadline_at`; every later claim generation and internal retry consumes the same absolute budget and cannot extend it.
- **R-089 — Propagate remaining budget through governed execution.** Foundation, model calls, Read-Only Query Tools, and Proposal adjudication receive a bounded budget derived from the Turn's remaining time and their own configured maximum.
- **R-090 — Reserve time for durable closure.** Component work must stop before the absolute deadline with a configurable L2 closure reserve so `Turn Terminal Outcome` CAS and terminal-event persistence can complete or retry without intentionally spending the entire budget inside L3.
- **R-091 — Do not invoke expired Turns.** A claim or reclaim that observes an elapsed execution deadline must not start Foundation and must instead compete for the unique timeout closure under the current claim fence.
- **R-092 — Keep queue and external deadlines independent.** Queue wait does not consume the execution deadline. Outbox attempts and Lab/Nexus Tasks own separate deadlines and continue independently after Turn timeout.
- **R-093 — Provide explicit user cancellation of the current Turn.** While the current user-triggered Turn is queued or running, Portal must expose a normal chatbot-style Stop action that requests cancellation of that Turn; disabling the composer or closing the live stream is not cancellation.
- **R-094 — Target cancellation by stable Turn identity.** The admission response must expose the server-assigned `turn_id` additively, and cancellation must address that exact Turn rather than a mutable session-level notion of "current" that could race with a newer Turn.
- **R-095 — Make cancellation durable and server-authoritative.** A successful cancellation request must be recorded by L2 and participate in the unique Turn-closure protocol. Browser navigation, SSE disconnect, or a local `AbortController` must neither impersonate nor be required for backend cancellation.
- **R-096 — Scope cancellation as a separately releasable cross-repo product slice.** Agent Service owns Turn identity, cancellation admission, execution interruption, durable closure, and event projection; Portal owns Stop/Stopping interaction and projection. The slice requires its own compatibility, rollout, rollback, and root Production PRD update, while Lab Service and Lab Task cancellation remain out of scope.
- **R-097 — Linearize cancellation and Proposal acceptance by first commit.** User cancellation and effectful Proposal acceptance must serialize through the same L2-owned Turn/effect authority. If cancellation commits first, the Turn can accept no later Proposal; if Proposal acceptance commits first, its Workflow Fact changes and Outbox Commands remain committed while cancellation stops only the remaining Agent execution and output.
- **R-098 — Terminalize accepted user cancellation immediately.** A successful cancellation transaction must atomically persist the unique cancellation `Turn Terminal Outcome`, append its terminal projection, and revoke all later Proposal, durable-output, and broadcast authority for that Turn. Cooperative model/tool interruption and resource cleanup follow asynchronously and cannot delay the authoritative outcome or introduce a `cancelling` Turn state.
- **R-099 — Project cancellation as a distinct non-error terminal event.** The accepted cancellation transaction must append one persisted `turn_cancelled` Session Event for SSE, replay, and Portal convergence. It must not reuse `turn_failed` or trigger failure-specific workflow/UI behavior; payload fields beyond the existing lifecycle identity/correlation contract remain provisional until their Field Necessity Records pass review.
- **R-100 — Authorize cancellation by initiator or current execution capability.** A Turn Initiator who still has Session `CHAT` may cancel their own user-triggered Turn. A current Session member with `EXECUTE` may cancel any cancellable Turn in that Session. An observer may therefore cancel their own Turn but cannot cancel another member's Turn; former and non-members cannot cancel.
- **R-101 — Retain a trusted Turn Initiator reference for authorization.** Durable admission of a user-triggered Turn must bind it to the authenticated initiating Principal so L2 can distinguish self-cancellation from privileged peer cancellation without trusting Portal input, focus ownership, event text, or trace data. The exact storage field name/type and any external exposure remain provisional.
- **R-102 — Limit user-facing cancellation to user-triggered Turns in v1.** Portal Stop and the user cancellation API accept only Turns admitted from trusted `source=USER`: currently `USER_MESSAGE`, `FORM_CONFIRM`, and genuine user `DECISION_RESPONSE`. Deterministic facts already committed by Input Admission remain valid when the later Agent continuation is cancelled.
- **R-103 — Keep System-Triggered Turns outside user cancellation.** Turns admitted from MQ, scheduler, or reconciliation—including `TASK_TERMINAL`, `DECISION_EXPIRED`, and reconciliation continuations—must reject the user cancellation API and rely on their deadline/watchdog/recovery policy. Operational shutdown or administrator recovery is a separate internal control. L2 assigns trusted source/kind; current expiry paths misrepresented as `DECISION_RESPONSE/USER` must be corrected before eligibility enforcement.
- **R-104 — Retain only durable completed output on cancellation.** Portal and replay retain already persisted `text_done`, `reasoning_done`, and completed tool-result segments, discard the currently unfinished emit-only text/reasoning/tool-call fragment, mark open execution presentation interrupted, and render a non-error cancellation marker.
- **R-105 — Do not add a partial-output draft in v1.** `turn_cancelled` carries no partial text snapshot, and v1 adds no durable response-draft/stream-accumulator persistence solely to preserve unfinished tokens. Such retention requires a separately approved consumer, consistency contract, flush/fencing design, and write-volume acceptance.
- **R-106 — Use an exact-Turn authenticated cancellation endpoint.** User cancellation uses `POST /sessions/{session_id}/turns/{turn_id}/cancel` with an empty request body. L1 derives the cancellation actor only from the authenticated Principal and passes the exact Session and Turn identities to L2; it accepts no client-supplied actor, initiator, reason, or mutable `current` selector.
- **R-107 — Return stable Turn identity from every eligible admission receipt.** The existing HTTP 202 receipts for user message, form confirmation, and genuine user decision response must each add the server-assigned stable Turn identity produced by the committed Input Admission transaction. The proposed JSON spelling is `turn_id`, but field spelling remains provisional until the API contract freezes.
- **R-108 — Make cancellation completion synchronous with Persistence Closure.** The cancellation endpoint returns HTTP 200 only after the L2 transaction has either committed the unique cancellation `Turn Terminal Outcome` or observed the already committed `Turn Terminal Outcome` winner. It must not return HTTP 202 for a merely requested cooperative interruption.
- **R-109 — Make cancellation retries idempotent and semantically explicit.** A successful response has exactly two semantic dispositions: `cancelled` when this or an earlier request durably cancelled the Turn, and `already_terminal` when completion, failure, or timeout had already won and cancellation did not occur. Exact response field and enum spellings remain provisional.
- **R-110 — Keep the cancellation response narrower than the Turn Terminal Outcome contract.** The response must not duplicate the full typed `Turn Terminal Outcome`; Portal obtains user-visible terminal detail from the authoritative Session Event/history projection. An unknown Turn or Session/Turn mismatch returns 404, insufficient cancellation authority returns 403, and a known System-Triggered or otherwise ineligible Turn returns 409 without changing the Turn.
- **R-111 — Persist the winning Cancellation Actor internally.** The transaction that first commits a cancellation `Turn Terminal Outcome` must also retain a trusted reference to the authenticated Principal whose request won. This Cancellation Actor is distinct from the Turn Initiator, is never supplied by Portal, and cannot be overwritten by a duplicate cancellation request. The exact column name, identifier encoding, and audit access mechanism remain provisional.
- **R-112 — Keep cancellation audit metadata minimal and private in v1.** The Cancellation Actor reference is internal durable audit data and is not added to `turn_cancelled`, SSE, snapshot/history projection, or the cancellation HTTP response. V1 accepts no free-text reason and persists no separate cancellation-source or `cancel_requested_at` field: the authenticated endpoint, cancellation `Turn Terminal Outcome` kind, and outcome commit timestamp already answer those questions unless a later named consumer proves otherwise.
- **R-113 — Give the absolute execution deadline precedence after its cutoff.** L2 evaluates cancellation eligibility against an authoritative clock at the `Turn Terminal Outcome` decision. Before `execution_deadline_at`, cancellation may commit normally; at or after the deadline, timeout is the only eligible outcome kind, the cancellation path may help persist or observe that outcome and returns the `already_terminal` semantic disposition. Equality belongs to timeout, watchdog scheduling delay cannot relabel timeout as cancellation, and committed Proposal/Command lifecycles remain unchanged.
- **R-114 — Make one Domain Pack expose two capability-limited faces.** A Domain Pack is one coherent, versioned domain module whose Manifest binds an L3-facing Domain Agent Definition and an L2-facing Domain Proposal Policy. It is an orthogonal extension boundary, not an L3-only plugin or a new architectural layer.
- **R-115 — Keep probabilistic domain composition on the Agent face.** The Domain Agent Definition supplies typed Domain Step Definitions, model-facing schemas, prompts, context contributors, classified Pure/Query/Proposal tools, evidence interpretation, and optional user-facing finale narration over reviewed Agent Foundation and Experiment Workflow Kit capabilities. It cannot replace the Kit-owned macro topology, return an arbitrary compiled graph or raw executable node, own Workflow Facts, authorization, transaction, outbox, or external-command execution, or supply facts or sections to the deterministic Summary Document.
- **R-116 — Keep deterministic domain adjudication on the Policy face.** The Domain Proposal Policy evaluates typed domain intent against an immutable current-fact/Workflow View and produces a pure typed policy decision or deterministic transition plan for L2. L2—not the Policy implementation—owns current-fact loading, global capability checks, CAS/locking, transaction execution, aggregate mutation, Session Event append, and Outbox creation.
- **R-117 — Bind both faces to one domain/version at composition.** The Composition Root must select and validate both faces from the same trusted installed Manifest and contract ranges. The Agent and Policy faces may share neutral domain contract types but cannot import one another's layer implementation or receive repositories, transaction handles, raw clients, or generic service containers.
- **R-118 — Add domains without editing domain-neutral cores.** Registering Biology or another Domain Pack may add pack code and deployment composition, but cannot require a domain conditional, enum member, graph edge, schema branch, or concrete import in Agent Foundation or generic L2 orchestration/adjudication code.
- **R-119 — Use one domain-neutral Agent Runtime Port from L2 to L3.** The L2 Turn Host invokes L3 only through one Turn-scoped execution contract. The Port accepts an immutable invocation request and returns typed runtime output plus exactly one `Foundation Execution Outcome`; exact method and field shapes remain provisional. L2 maps that result and its current authoritative facts into the durable `Turn Terminal Outcome` through its own Persistence Closure.
- **R-120 — Hide Agent composition behind the Runtime Port.** L2 cannot select a specialist Agent, graph, node, prompt, model-visible tool, or Step capability. The prevalidated Agent Foundation, Experiment Workflow Kit, and Domain Agent Definition compose execution inside L3, while the Composition Root binds them using authoritative workflow domain identity rather than model or client choice.
- **R-121 — Keep invocation and Proposal seams distinct.** The Agent Runtime Port is the L2-to-L3 execution entrance. During that execution, the Proposal Port is the governed L3-to-L2 request/reply seam; its L2 implementation uses the selected Domain Proposal Policy. The Domain Pack binds both faces but does not create a third layer or a direct Agent-to-Policy call.
- **R-122 — Deepen rather than replace the existing single-entry shape.** The redesign may preserve the conceptual current `Runtime.invoke(ctx, turn)` boundary while replacing its contaminated context, raw dependencies, graph routing, streamed mutation events, and iterator-exhaustion closure inference with the governed contracts above.
- **R-123 — Use three reuse zones without adding architectural layers.** Cross-domain reuse is divided into Agent Foundation execution mechanics, an Experiment Workflow Kit for shared experimental-product behavior, and Domain Packs for irreducible scientific semantics. These are responsibility and reuse zones inside the existing L1-L4 taxonomy, not a second layer stack.
- **R-124 — Keep the Experiment Workflow Kit layer-aligned and authority-free.** The Kit may provide neutral contracts plus reusable L2 deterministic workflow components and L3 Agent components, but it is not a runtime intermediary, persistence authority, Domain Pack superclass, plugin system, or workflow DSL. Generic L2 retains transaction and Workflow Fact authority; Agent Foundation retains governed execution authority.
- **R-125 — Standardize the v1 Experiment Workflow Template.** V1 supports one Experiment Objective followed by one Experiment Plan containing an ordered serial sequence of domain-defined Experiment Steps. After the final Step's results are confirmed, the Experiment Workflow Kit completes the Plan and gates and orchestrates the Summary Document through the existing compatible report path. In Chemistry v1, `Summary Document` is the cross-domain architectural name for the existing Chinese/English ELN Word report: it is a deterministic, AI-free aggregation of confirmed experiment data, with the existing authorization, all-results-confirmed gate, placeholder, and optional-enrichment behavior preserved.
- **R-126 — Standardize each v1 Experiment Step cycle.** Every Experiment Step follows the same conceptual three-phase cycle: Preparation/Parameter Design, Experiment Execution, and Result Analysis. When the Step requires system-generated execution parameters, Preparation/Parameter Design returns a typed execution-bearing design and the existing human Confirmation Gate must pass before execution. When it does not, including a manual Step without robot-execution parameters, the phase returns typed `not_required`, creates no synthetic parameter form or confirmation surface, and proceeds to the existing human completion and evidence-input flow. Result Analysis closes the current Step cycle and determines the next permitted plan-progression request.
- **R-127 — Keep Step science domain-owned.** A Domain Pack defines Step kinds, schemas, prompts, parameter-design behavior, evidence meaning, result-analysis behavior, Proposal payloads/policy, and typed deterministic report data/section contributors derived from confirmed Workflow Facts and Physical Execution Facts. It must compose the shared Experiment Workflow Kit and Agent Foundation capabilities rather than reimplementing their lifecycle, model/tool loop, stream, context-budget, deadline, cancellation, or closure mechanics. Agent-generated finale narration is separate and cannot supply report facts, alter the ELN, or replace it.
- **R-128 — Execute one planned Step at a time in v1.** The Experiment Workflow Kit and generic L2 progression rules allow at most one active Step execution in a Plan. A later Step cannot begin execution before the current Step has reached its authoritative result-analysis/progression decision.
- **R-129 — Do not build a general plan-topology engine in v1.** Parallel steps, DAG routing, nested plans, and a general dynamic workflow language are outside the initial contract. Serial assumptions must be localized in the Experiment Workflow Kit and expressed through stable Plan/Step identities and versioned contracts so a separately approved future contract can add another topology without changing Agent Foundation or contaminating Domain Packs with shared mechanics.
- **R-130 — Gate the common substrate on second-domain leverage.** Before v1 contracts freeze, the Biology validation slice must use the same Agent Foundation public SPI and Experiment Workflow Template/Kit contracts without reimplementing model/tool governance, context budgeting, streaming, Proposal plumbing, execution lifecycle, Objective/Plan progression, conditional Step-design confirmation, execution correlation, result-analysis progression, or Summary Document gating and orchestration.
- **R-131 — Promote shared behavior by semantics, not code resemblance.** Safety and authority invariants enter the owning neutral core immediately. Other behavior moves from a Domain Pack into the Experiment Workflow Kit only after Chemistry and Biology demonstrate identical semantics and the extraction removes duplication without domain conditionals; prompts, action schemas, evidence meaning, and scientific transitions remain domain-owned even when their code looks similar.
- **R-132 — Freeze a confirmed Plan's structure.** Confirmation makes the Plan's ordered Step identities, order, execution ownership/type, and domain Step-definition references immutable. Only dedicated L2 progression transitions may change progression data; an Agent, Domain Pack, generic update method, replay, or later confirmation cannot overwrite the confirmed structure in place. Exact persistence fields remain provisional.
- **R-133 — Model same-Step rework as a new Trial.** Repeating execution or analysis for the current Step creates a new Trial under the same stable Plan and Step identity. It does not insert another planned Step, mutate the confirmed Plan, create a Turn Attempt, or reopen an earlier Turn; any changed execution parameters must pass the applicable confirmation policy again.
- **R-134 — Keep Plan Revision outside v1.** V1 does not support adding, removing, replacing, or reordering Steps after Plan confirmation and must reject such structural mutation rather than editing the confirmed Plan. A future separately approved Revision capability must use a new confirmed Plan identity, preserve completed Step/Trial history, and affect only remaining work.
- **R-135 — Do not predesign Revision infrastructure.** This refactor adds no Revision API, lineage view, active-revision selector, `revision`/`supersedes` persistence field, or Revision-specific Portal behavior. Exact Revision and carry-forward semantics remain outside the v1 contract until a named product use case is approved.
- **R-136 — Give the Experiment Workflow Kit exclusive ownership of the v1 macro topology.** The Kit owns and enforces Objective, confirmed serial Plan, Step-cycle phase order, applicability and enforcement of confirmation gates, Trial lifecycle, serial progression, Plan completion, and Summary Document gating and orchestration. A Domain Pack cannot supply, compile, or execute a replacement macro workflow graph.
- **R-137 — Represent domain Steps as typed aggregate definitions.** A Domain Agent Definition supplies coherent Domain Step Definitions composed from narrow phase-scoped capabilities for Step identity/language, Preparation/Parameter Design, Execution Intent, and Result Analysis, including a typed loop intent when the analysis can request another Trial. Exact Python methods, fields, and generic parameters remain provisional until the Biology slice exercises them.
- **R-138 — Keep Step capabilities on the correct Domain Pack face.** L3-facing Step capabilities provide probabilistic design/analysis behavior, typed schemas, context, and Intent Payload construction without persistence or command authority. Corresponding pure preconditions and deterministic transition decisions remain on the L2-facing Domain Proposal Policy, bound by neutral versioned Step identity/contracts rather than one cross-layer executable object.
- **R-139 — Keep robot and manual Steps on one lifecycle.** Execution ownership/mode is Plan Step data, not a different graph topology. A robot Step supplies typed external-execution intent that reaches the Proposal/outbox path. A manual Step supplies a typed human-completion and evidence-input contract and, when it needs no system-generated execution parameters, returns Preparation/Parameter Design `not_required` without a synthetic parameter form or confirmation surface. Both use the same conceptual phase order, Trial, Result Analysis, result-review, progression, and Summary Document gating/orchestration mechanics; confirmation is required only when the Step exposes an execution-bearing design.
- **R-140 — Limit Step-local loops to the current Step and Trial lifecycle.** Result Analysis may produce a typed scientific loop intent, and the L2-facing Domain Proposal Policy may supply the corresponding pure Step Loop Policy and retry transition inputs. The Kit and generic L2 own Trial creation, idempotency, confirmation, and progression. A loop cannot edit Plan topology, advance another Step, call persistence, or execute an external command directly.
- **R-141 — Avoid both giant callback records and independent phase registries.** A Domain Step Definition is a small typed aggregate referencing deep phase capability modules. It must not expose a long optional-callback list, `Any`/`object` payloads, mutable runtime state, a generic service locator, or separately registered phase maps that require runtime casts to reconstruct one Step.
- **R-142 — Version topology evolution separately from scientific variation.** New scientific behavior within the fixed cycle is added through new Step Definitions or phase-capability implementations. Parallelism, DAGs, nested Plans, additional macro phases, or another workflow shape require a separately approved versioned Experiment Workflow Template contract rather than optional raw-graph hooks on a Step Definition.
- **R-143 — Govern future Agent capabilities through typed families.** Memory, Skills, and MCP share one demand-gated admission, declaration, capability-grant, provenance, and audit policy but do not share a generic runtime invocation interface. Foundation, the Workflow Kit, Domain Packs, Skills, and the model receive no `invoke(name, payload)` extension bus or generic plugin/service locator.
- **R-144 — Grant extensions at the narrowest composition seam.** The Composition Root resolves only trusted configured providers, validates their declared contracts and permissions, and grants each approved typed capability to the narrowest Foundation, phase, Domain Step, or Skill consumer. Domain code receives no raw store, MCP session/client, credential, network client, persistence handle, or provider registry.
- **R-145 — Govern future Domain-defined Skills as typed contributions.** If a Domain Skill capability passes its Demand Gate, a trusted statically installed Domain Skill Definition may contribute domain instructions, prompt/context blocks, Pure/Query/Proposal Tool declarations, and bindings to declared domains, Step kinds, and workflow phases. It cannot provide a replacement macro graph, arbitrary middleware, workflow policy authority, persistence, external-command execution, or undeclared capabilities. The v1 conformance fixture adapts directly into neutral contribution seams and does not freeze this feature-specific production interface.
- **R-146 — Prevent Skill capability amplification.** A Skill executes only with the intersection of current Principal capabilities, Domain Pack declarations, provider grants, phase policy, and tool classification. A Skill cannot introduce a new permission, become an authorization source or Domain Proposal Policy, delegate a capability it did not receive, or change a tool's effect category.
- **R-147 — Treat any future approved MCP capability as an adapter mechanism, not a trust boundary.** If an MCP use case passes its Demand Gate, Domain Packs or Skills may declare typed operation requirements and domain interpretation, while the Composition Root resolves configured providers and validates every operation separately for schema, stable identity, source, domain/phase allowlist, required capabilities, provenance, deadline, audit, and effect category. No MCP server is admitted wholesale as trusted or read-only, and the v1 fake Query fixture does not freeze a production provider-resolution interface.
- **R-148 — Route any approved MCP effect through existing authority seams.** An admitted MCP read operation becomes a governed Query capability; model-requested mutation can become only a typed Proposal intent; an operation that actually changes externally owned state is never model-visible and may run only as an outbox-executor adapter after commit. Domain/Skill code never receives a generic MCP `call_tool` surface. V1 proves the Query path and rejects effectful exposure without implementing production mutation or command integrations.
- **R-149 — Make general Agent Memory a Foundation-governed capability family.** A future approved Memory provider is selected through composition and exposed through typed read/write seams while its concrete Store remains behind an L4 port. Domain Packs and Skills may declare memory requirements and provide domain extraction, rendering, or retrieval semantics, but cannot own or access the raw Store.
- **R-150 — Admit Memory reads as lower-authority context.** Memory retrieval produces provenance-bearing, scoped, time/freshness-aware Context Blocks or governed Query results subject to Foundation ranking, deduplication, protection, and token budget. Memory cannot overwrite, contradict, or become an alternative authority for Workflow Facts, Physical Execution Facts, current permissions, or durable Turn/Plan/Trial lifecycle state.
- **R-151 — Separate trusted summaries from model-selected Memory writes.** The bounded v1 rolling-conversation-summary slice uses a trusted session projection: it produces and stores the projection before source Turns leave the 50-event window, and Foundation loads it before recent verbatim events. This does not create a general Memory provider or model-selected write path. A model, Skill, or domain Agent that later chooses to persist Memory must submit a typed Memory Proposal for scope, permission, content-policy, concurrency, provenance, and idempotency adjudication; it cannot write the Store directly.
- **R-152 — Keep capability implementation demand-gated.** Agreeing these typed extension boundaries does not add a production Memory Store, general Skill runtime, MCP gateway, dynamic provider loader, or untrusted executable extension to Foundation v1. Each implementation still requires a named consumer/use case and measurable gate; dynamic or untrusted Skill/MCP execution additionally requires the separately approved process/sandbox threat model.
- **R-153 — Ship only validated neutral capability seams in v1.** Production refactor scope may implement typed context contribution/assembly, classified Pure/Query/Proposal Tool hosting, Manifest capability requirements and grants, and default-deny Composition Root validation only when migrated Chemistry or the Biology validation slice exercises those neutral mechanisms before contract freeze. It must not freeze or implement feature-specific `MemoryProvider`, `SkillRegistry`, or `MCPGateway` interfaces merely to claim future support.
- **R-154 — Validate optional-capability geometry with internal conformance fixtures.** Non-production conformance work must include one static Biology-phase contribution shaped like a Domain Skill and adapted directly into neutral prompt/context/tool seams, one allowlisted fake MCP read operation adapted through the governed Query seam, and one in-memory memory-like context contributor. Together they test contribution composition, capability intersection, Principal/deadline/audit/provenance propagation, startup rejection, context scope/freshness/budgeting, and authoritative Workflow Fact precedence.
- **R-155 — Keep conformance evidence distinct from product capability approval.** Capability Conformance Fixtures are disposable test components that exercise neutral seams; they are not production providers, public extension contracts, Demand-Gated Capability approvals, or evidence that a feature-specific provider interface is frozen. Production composition must not discover or enable them.
- **R-156 — Exclude optional-provider and cross-repo expansion from v1.** The capability-validation scope adds no persistent or vector Memory Store, retention/deletion policy, generic Memory write pipeline, Skill discovery/installation/hot-loading/registry/sandbox, generic MCP discovery/gateway/dynamic activation/credential lifecycle, database migration, or Portal/API/SSE/Lab Service/BIC-shared-types change.
- **R-157 — Defer workflow-lifetime behavior binding to a production gate.** V1 adds no Workflow Behavior Binding, Behavior Target persistence, legacy backfill, cohort admission, or version-retention machinery. Before the first real production deployment where experiments must survive Agent Service releases, ADR-0035 must define immutable behavior identity, rollback, version retention, in-flight Proposal/Command handling, migration, and history semantics.
- **R-158 — Validate exact deployment composition.** The Composition Root must bind one compatible Domain Pack Manifest, Domain Agent Definition, Domain Proposal Policy, Experiment Workflow Template/Kit, and Foundation adapter set for each configured route. Both Domain Pack faces come from the same Manifest version, and startup rejects missing, duplicate, or incompatible components.
- **R-159 — Merge migration code to main behind disabled routing.** New implementation slices land incrementally on `main` and remain disabled for ordinary workflows until the combined cutover gate passes. A long-lived migration branch is not the delivery mechanism.
- **R-160 — Forbid runtime fallback and dual authority.** A named internal validation workflow cannot change implementation by Turn, Step, Plan shape, model output, unsupported branch, or failure. The migration cannot effectfully shadow-run, dual-write Workflow Facts or Session Events, accept the same Proposal twice, or create a second external-command path.
- **R-161 — Quiesce or reset old-path work before cutover.** After the combined gates pass, every unfinished old-path workflow on the bench and at field sites must complete or be explicitly reset before the default route changes.
- **R-162 — Switch once and delete legacy.** The release performs one default-route switch, verifies smoke and rollback evidence, then removes legacy routing and code. V1 does not maintain online cohorts, persisted legacy bindings, a binding drain, or an indefinite compatibility flag.
- **R-163 — Use exactly two Model Capability Levels in v1.** Foundation defines the centrally governed, provider-neutral levels `light` and `complex`; no other level exists in v1. A level is the complete model requirement exposed to Agent graphs. Graphs do not declare arbitrary capability bags, model roles, providers, model identifiers, endpoints, or deployments, and Domain Packs cannot add levels.
- **R-164 — Require every hosted Agent graph to declare one level.** Each Foundation/Kit-hosted Agent graph declaration selects exactly one Model Capability Level. Foundation alone maps that level to an approved concrete model when the graph is invoked. A missing, unknown, or unmapped level fails closed before model invocation. Portal/client payloads, prompts, model output, graph nodes, Domain code, and Skills cannot select or change the provider, model, or level at runtime.
- **R-165 — Permit governed model evolution behind a stable level.** Foundation may change the approved concrete model mapped to a Model Capability Level only after the tests and rollout gates required for that level pass. The mapping gate reuses the Framework Ruling's DashScope/Qwen live probe for structured output, tool calling, streaming, usage, timeout, cancellation, and error behavior. This permission does not make model outputs deterministic or allow an unreviewed “latest model” selector.
- **R-166 — Record actual model use as Turn-correlated provenance.** Foundation must emit structured Model Invocation Provenance for every actual model call and correlate it to the Turn. The minimum record must identify the hosted graph, its declared Model Capability Level, and the concrete provider/model used for that call. Provider revision, mapping-policy version, behavior-config fingerprint, timing, usage, outcome, and other candidate fields are included only when available and accepted by consumer-driven Field Necessity review; credentials, secret endpoints, and raw prompts/responses are excluded by default.
- **R-167 — Keep model provenance internal in v1.** L2 retains the approved compact Turn-correlated audit record while full call detail may remain in telemetry. V1 adds no model-selection or model-provenance field to Portal APIs, SSE, snapshots, Lab Service, or BIC-shared-types, and model provenance cannot become Workflow Fact authority or alter Proposal adjudication.
- **R-168 — Demand-gate exact model pinning.** A domain that requires the same exact provider/model revision for regulatory validation or scientific reproducibility must obtain a separately approved policy with availability, deprecation, retention, rollback, and failure semantics. Foundation v1 does not infer exact-model pinning from Domain Pack version or an ordinary Model Capability Level declaration.
- **R-169 — Migrate L3 through DL-01 vertical slices.** The refactor must not build and freeze a complete horizontal platform before exercising product behavior, and it must not use one big-bang cutover. Each implementation child complies with DL-01: one to three engineer-days, one pre-committed binary acceptance check, and one rollback boundary around the minimum L1-L4 path needed for the behavior.
- **R-170 — Make the complete Chemistry happy path the first validation target.** The first production-shaped new-architecture slice must traverse the recorded baseline lifecycle from Objective through confirmed Plan, its ordered Chemistry Steps and Lab round trips, Plan Completion, and the existing deterministic ELN Summary Document while preserving existing Portal, Lab, Session Event, snapshot, replay, conditional confirmation, and ELN semantics. This first slice runs through internal contract, golden, and live-bench harnesses behind disabled routing; passing only the happy path does not permit the default route to switch.
- **R-171 — Keep migration on main with one disabled new route.** New-path slices merge incrementally to `main`; ordinary workflows continue through the old default while named internal validation runs use the disabled new route. The design rejects a long-lived migration branch, effectful dual-running, and behavior selection controlled by model output or workflow branches.
- **R-172 — Migrate remaining Chemistry behavior before portability and cutover gates.** After the complete happy path passes, remaining baseline Chemistry branches and recovery behavior move in independently verifiable slices behind disabled routing while contracts remain provisional. Unsupported behavior fails the named validation run and never causes per-Turn or per-Step legacy fallback.
- **R-173 — Use Biology as a pre-freeze portability gate.** Once the production-shaped happy path and remaining migrated Chemistry behavior exercise the provisional Agent Foundation public SPI, Experiment Workflow Template/Kit, and Domain Pack seams, the non-production Biology slice must traverse the same lifecycle and may force those provisional contracts to change. It does not become a production rollout or authorize an external contract change.
- **R-174 — Freeze the portability-tested contracts and revalidate all Chemistry before cutover.** Only after the Biology slice passes may the Agent Foundation public SPI, Experiment Workflow Template/Kit contracts, and Domain Pack contract freeze as v1. Every migrated Chemistry slice, not only slices affected by a Biology-driven change, must then pass its contract, golden, architecture, failure-injection, and applicable live-bench gates against those frozen contracts. The default route cannot switch until every baseline Chemistry behavior is implemented and gated.
- **R-175 — Retire legacy L3 after the simplified exit gate.** Legacy L3 may be removed only after all required Chemistry behavior passes on the new path, the Biology and frozen-contract rerun gates pass, external compatibility is proven, rollback evidence exists, and no unfinished old-path workflow remains on the bench or at a field site because each has completed or been explicitly reset. The release then switches once and deletes legacy routing and code.
- **R-176 — Apply database compatibility discipline only to retained data.** The migration must first decide whether `a1` field data or `orin-tail` data must survive. If either must be retained, affected persistence changes use expand, migrate, contract until the retained records are verified on the new schema. If both may reset, reset-and-seed is allowed and no unconditional dual-version schema requirement remains.

## Acceptance Criteria

- [ ] The technical design contains an explicit ownership table for product workflow facts, physical execution facts, Session Events, and Agent execution state, with no field assigned to competing authorities.
- [ ] Every live or durable event kind is classified as a transient broadcast delta, durable Turn conversation/observation output, workflow-transition event, Input Admission event, or terminal event; tests prove `user_message_submitted` commits with Admission, `FormRequested` commits with its accepted pending-decision action, and only workflow-transition events require Proposal acceptance.
- [ ] No business aggregate snapshot is treated as authoritative after being serialized into Agent execution state or a checkpoint.
- [ ] Compatibility evidence covers semantic equivalence across Portal live, snapshot, and replay paths.
- [ ] Architecture tests fail when L3 imports L2/session implementations, accesses workflow persistence, or invokes external command adapters outside the approved effect boundary.
- [ ] Architecture tests fail when L4 adapters contain workflow transition policy rather than technical I/O translation.
- [ ] Current Y-4 graph topology and TLC/CC/FP/RE wiring are absent from the definition of the stable L1-L4 contract.
- [ ] Architecture documentation contains only one layer taxonomy and locates Agent Foundation, Domain Pack, and Proposal Boundary relative to L1-L4.
- [ ] An effectful L3 tool, Skill, or MCP integration cannot directly mutate workflow persistence or execute an external system command; it can only produce a typed Proposal.
- [ ] Accepted Proposals reuse one L2-owned deterministic transaction path for current-fact loading, authorization, CAS, aggregate mutation, event append, and outbox creation.
- [ ] No proposal-specific reducer, event log, or persisted workflow authority exists in parallel with the canonical L2 path.
- [ ] The initial implementation plan does not schedule long-term Memory, a general Skill registry, general MCP activation, or durable Agent threads without a separately reviewed demand-gate artifact.
- [ ] Every future-capability gate names its consumer and use case, defines a measurable outcome, identifies state/authority/safety constraints, and explains why existing workflow facts, Session Events, ordinary retrieval, or database-driven retriggering are insufficient.
- [ ] Outward Foundation contracts expose only BIC-owned framework-neutral types; tests reject `langchain*`, `langgraph*`, and `pydantic_ai*` imports or types in Domain Packs and L2-facing context, Proposal, result, stream, or runtime-event contracts.
- [ ] Inward provider-contract tests prove that every permitted public framework adapter satisfies the same BIC-owned model, tool, context/state, stream, and checkpointer contract without relying on private APIs or undocumented lifecycle metadata.
- [ ] A minimal Biology objective and one Biology expert Agent complete objective, plan, Preparation/Parameter Design with either a confirmed execution-bearing design or typed `not_required`, Proposal handling, simulated Lab dispatch and asynchronous progress, result confirmation, and deterministic report generation through production-shaped Agent Service ports.
- [ ] The passing Biology slice requires no Biology-specific import, conditional, enum member, prompt, schema, workflow edge, or report branch inside Agent Foundation or the Experiment Workflow Kit.
- [ ] The Biology validation introduces no required Portal, Lab Service, or BIC-shared-types change.
- [ ] The Agent Foundation public SPI, Experiment Workflow Template/Kit contracts, and Domain Pack contract remain provisional while the complete Chemistry happy path and remaining Chemistry slices exercise them; all three freeze only after Biology records every exercised contract field and pressure point, after which every migrated Chemistry slice passes again behind the frozen contracts.
- [ ] Before implementation, the task records a compatibility baseline and an inventory of externally observable Chemistry contracts across Portal, Lab Service, and BIC-shared-types.
- [ ] Every implementation child complies with DL-01 by recording a one-to-three-engineer-day estimate, owner, dependencies, rollback boundary, pre-committed binary acceptance check, and actual duration.
- [ ] Every migration slice passes contract/golden tests and the applicable live-bench scenarios against unchanged compatible versions of Portal, Lab Service, and BIC-shared-types.
- [ ] An adopt/adapt/drop table covers all externally visible additions from PR #94 and PR #136, including events, HTTP errors, snapshot/form fields, and task-parameter provenance fields.
- [ ] The compatibility baseline records ADR-0012's approved failure-path exception for every affected action: the current partial workflow-transition prefix and the target all-or-nothing Proposal trajectory. Any other intentional product behavior or required external-repo change remains separately scoped and approved.
- [ ] Architecture tests prevent direct imports or calls to state-changing Lab/Nexus command adapters from L3, Domain Packs, Proposal handlers, and ordinary tools.
- [ ] Transaction tests prove that an accepted effectful Proposal commits its aggregate change, ordered workflow-transition Session Events, and Outbox Commands atomically, with no external call before commit.
- [ ] Failure injection before Proposal commit exposes no workflow-transition fact, workflow-transition event, or command from that action; failure after commit exposes the complete ordered action even if the Turn later fails, while separately committed conversation/observation output remains valid.
- [ ] Failure-injection tests at claim, call, response, receipt, and broadcast boundaries produce one logical command, one workflow transition, and one replayable UI trajectory.
