# BIC Agent 协作流程契约（接口定义文件）v0 草案

## 状态

- 性质：**接口定义草案，待逐格审定后冻结接口层**；格式层（附录模板）为 v0，试点数据驱动演进，演进不破坏接口层。
- 设计原则：接口冻结 ≠ 格式冻结；每阶段出口必须二元；会话内不留孤知；agent 先出稿、owner 裁定（决策点必须"选项+推荐"，禁单一成稿）。
- 配套：`agent-team-workflow-v2-draft.md`（拓扑与原则）；本文件是其阶段模型的逐格契约展开。

## 阅读方式

每阶段一行五列：**入口**（什么工件触发）｜**执行者**｜**产物**（形式+存放位置）｜**二元出口**（过/不过可外部检验）｜**裁定人/超时**。

## 二、共享管线（三线在此汇合）

| 阶段 | 入口 | 执行者 | 产物 | 二元出口 | 裁定人/超时 |
|---|---|---|---|---|---|
| C1 实现 | fix：锚定 issue（必要时先裁定）；dev：任务链就绪 | 实现会话（brief 冷启动、worktree 隔离） | PR：squash 正文含 issue 号、具名测试（正/负/回归）、Depends-on、偏差申报节 | CI 全绿 ∧ 具名测试在 ∧ brief 要求逐条勾 | 大偏差升裁定（判据见回流）/ —— |
| C2 合并 | CI 绿的 PR | surface owner（CODEOWNERS） | review + 合并 | review 通过 ∧ squash 入 main ∧ 内容核验（issue 号在正文 + 具名测试在 main） | owner / review >2 工作日催办；**owner 待审 WIP ≤ N** |
| C3 部署 | main 前进 | 确定性脚本（`make update` / `ops/field/update.sh`）+ 部署锁 | 部署执行+健康门 | 目标环境健康门通过 ∧ 部署 sha 回填 issue | —— / 合批节奏由持锁 S1 定 |
| C4 复测验收 | 部署 sha 回填 + 飞书复测点通知 | **reporter**（fix）/ 意图发起人（dev） | 关闭评论（通过）或打回评论（带证据） | issue closed ∨ reopened-with-evidence（回 C1 或 F2） | 提出人 / >5 工作日标记"已部署未验证"入集中复测批 |
| C5 固化 | 事故类 issue 关闭前 | 托管 S1 | 守卫 PR（CI 检查/脚本拦停/文档）+ 教训条目 | 守卫合并 | —— / 事故后 24h 内 |

## 三、回流（实现/使用中发现文档与现实不符）

| 类型 | 判据 | 流程 | 二元出口 |
|---|---|---|---|
| B1 事实型 | 修正**不改变任何人的选择空间**（文档写错了现实） | agent 直接提修正 PR（附证据），owner 仅 review | 修正 PR 合并 ∧ 与原 issue 互链 |
| B2 裁定型 | 修正**改变选择空间**（设计意图变化） | 先 owner 拍板（选项+推荐格式），后动文档 | 裁定在案 ∧ 文档 PR 合并 |

## 四、会话生命周期（横切所有阶段）

spawn（凭 brief 冷启动）→ 工作 → 收尾（**单行 done + 核退出码**；长版 FACTS/JUDGMENT 落 issue 评论）→ 归档销毁。
铁律：**会话内不留孤知**——值得留的结论必须在 done 前落 issue/文档，否则视为没发生。

## 五、横切约束汇总

1. 并行纪律：worktree 隔离；同文件互 ack；高交集串行；结构性变更先落热区看板宣告，S1 派发前必查；
2. 部署纪律：共享环境有锁；确定性归脚本，LLM 只在拦停点介入；
3. 验证纪律：author≠verifier；内容核验不信自述；FACTS/JUDGMENT 分离；
4. 裁定纪律：rule 5 冲突挑明选一；裁定当天入 PRD/issue；裁定必须来自有权者；
5. 供给纪律：agent 先出稿；决策点选项+推荐；owner 待审 WIP 上限。

## 附录：格式模板 v0（版本化演进层，迭代不破坏上表接口）

### A1 fix issue 模板
```
## 现象（原话+截图）
## 复现入口（环境 / session 链接 / 操作序列）
## 一手证据（file:line / 日志 / DB —— 归流时补）
## 定性方向（分叉假设，不预设结论）
## 修复要求 & 二元验收
Reported-by: @<github-handle>   Refs: #...
```

### A2 dev 意图模板
```
## 意图（发起人原话）
## 目标场景 / 用户
## 约束（必须满足） / 非目标（明确不做）
## 期望时间窗（可空）
Initiated-by: @<handle>
```

### A3 brief 模板（派发实现/调查会话）
```
# 任务：<issue号> <一句话>
任务书指针（issue/评论/PRD 条目）｜仓+worktree+分支
## 先调查（如需）：结论落 issue 评论
## 实现口径（已裁定勿重议项显式标注）
## 二元验收：具名测试 + 门禁 + 内容核验
## 并行知会：同仓在跑会话 + 互 ack 要求
## 收尾：done 单行短摘要+核退出码；长版落 issue
```

### A4 收尾模板（会话 done 前落 issue 的长版）
```
## FACTS（可核验：sha/计数/证据链/未做之事显式列出）
## JUDGMENT（我的判断，可能有误）
```

### A5 ops 方案 PR 模板
```
## 摘要（结论先行）
## Facts（现状勘察，一手、只读）
## 方案（含回滚、预写二元验收清单、不可逆步骤标注）
## 开放问题（需拍板项逐条列）
```

## 待试点回填的数值参数

- owner 待审 WIP 上限 N（建议初值 3）；复测超时天数（建议 5 工作日）；裁定升级天数（建议 2 工作日）；集中复测批节奏（建议每周一次）。
