# Demo 测试剧本（供 T 系列测试员会话使用）

来源：飞书《Demo剧本详细版》（wiki YWDxwEQKEitmgtkuWBTcBDAFn7g，2026-07-08 蒸馏）。
测试员职责：像真实化学家一样通过浏览器走产品，逐步截图取证，**对照本剧本的预期行为**
发现偏差，写成 findings（不建 issue —— S1 统一查重建档）。

## 黄金主链路（T-main 的剧本）

1. **注册/登录**：portal `http://localhost:5174`，注册新用户（用户名前缀 `tmain-`）。
2. **开场白**（逐字使用）：
   `我要做个过柱，反应式为：C#CC(C)(O)C.BrC1=CC=C(I)C(COC)=C1>>BrC2=CC=C(C(COC)=C2)C#CC(C)(O)C，目标纯度95%，目标质量80%， 主反应物投料量15mg`
3. **实验目标**：agent 应解析反应式、投料表（Mind 返回目标重量），用户确认。
   预期：不再追问已给出的信息（纯度/收率/投料量）；表单预填；叙述第二人称、指引明确。
4. **流程编排**：默认 TLC/过柱/组分收集/旋蒸 四步全为机器人任务；确认。
   预期：确认后叙述以"我开始第一步"口吻收尾并指向表单；无"移交专员"。
5. **TLC 参数设计**：目标 Rf 0.5±0.1（窗口 0.4–0.6，**这个窗口配 mock 会先失败一次再成功**）；
   展开剂推荐（bench mock 推 PE:EA 5:1）。样品管：在右侧面板选择（demo 摆位：纯品 第1排第1个、
   粗品 第4排第1个，L2-样品管盒01；bench 用维护模式先填格再选）。
   预期：rxn 预填；只让用户补无来源项；样品管引导去面板选择而非聊天要 ID。
6. **物料准备/下发**：确认参数 → 下发。预期：下发成功（若样品管有效）；总览进入执行态。
7. **TLC 执行**：**demo 预期 retry 恰好 1 次**（第一次推荐条件不落窗，第二次成功）——
   bench mock 是 forced_retry 剧本（0.25→0.25→0.51），窗口 0.4–0.6 时第 3 次成功也算通过；
   关键断言：中间失败尝试静默重试、聊天只出现一次结果确认；结果含点板识别呈现。
8. **接受 TLC 结果** → 预期链路推进到过柱（CC 表单激活）。⚠️已知 bug #5 修复未生效前会停 —— 记录现象即可。
9. **过柱**：50mg 样品、12g 硅胶柱、洗脱按推荐；样品柱在面板选择。执行完组分判断结果可修改。
10. **组分收集**：简单确认（FP 面板：上游 CC 结果 + 容器/试管分配预填）。
11. **旋蒸**：溶剂体系=过柱洗脱体系、溶剂量=产物管体积加总（FP 确认应自动预填 RE 依据）；简单确认。
12. **导出报告**：全部结果确认后，结果确认面下载 ELN Word 报告（zh/en）。

## 挑战/异常剧本（T-challenge 的方向，不穷举）

- **输入挑战**：无效/畸形 SMILES；缺投料量；中途改目标（"纯度改成99%"）；中文/英文混输；
  一句话给全信息 vs 挤牙膏式分次给；答非所问；用"过柱"等目标词试探身份混淆。
- **流程挑战**：确认前刷新页面（状态应还原一致）；表单手改后发消息（改动不应被回滚）；
  在错误阶段要求跳步（"直接旋蒸"）；重复确认；失败后按叙述指引重试是否真的可行。
- **实验室查询**（剧本"实验室管理查询"节）：查任务参数/日志/异常、查物料（有什么在哪里）、
  查设备状态占用、查汇总统计 —— 核对回答与 DB 事实（talos-postgres:5433）。
- **UI 一致性**：thinking vs 聊天语域；工具卡有无悬挂"待处理"；总览/监控状态是否如实；
  左右面板状态同步；失败态呈现。
- ⚠️ **禁止**：真正下发机器人任务（dispatch）——那是 T-main 的独占资源；挑战流全部止步于
  参数确认之前，或走查询/表单/刷新类操作。

## 严重度分级（与台账一致）

`P0-链路断`（主链路无法推进/数据损坏）· `P1-功能`（行为错误/诚实性/无恢复路径）·
`P2-话术`（语气/语域/指引/语言）· `P3-UIUX`（布局/闪跳/观感）

## Bench 铁律（违者会毁掉用户和其他会话的工作）

- портal 在 **:5174**；agent/lab DB 都在 **talos-postgres:5433**（bic-postgres:5432 同名库是假的）。
- **绝不调用** `POST :8800/reset`（会清掉所有人的会话）；**绝不** TRUNCATE/写 DB。
- **轮末 lab reset 是标准程序（用户指令 2026-07-08）**：T-main 每完成一整轮主链路后，
  必须重置 lab 数据：
  `curl --noproxy '*' -s -X POST http://127.0.0.1:8192/admin/reset-to-test-data -H 'Content-Type: application/json' --data-raw '{"robot_id":"talos.001"}'`
  注意：reset 会触发机器人例程（忙数分钟）——下一轮 dispatch 前必须轮询
  `GET :8192/robots/idle` 直到非空。reset 前后各 `dispatch send` 通知 root（用户可能在手测，
  root 需要知情转达）。**轮中不 reset；T-challenge 永远不 reset。**
- 每个测试场景用**新注册用户 + 新会话**；不碰别人的会话。
- 浏览器一律 Playwright（node 脚本放 scratchpad，chromium headless）；每步截图。
- BE 处于 no-reload 冻结态：你测的是当前运行版；已修未生效清单见 root 通知，发现疑似已知问题照记，S1 负责去重。

## 通知降噪（2026-07-08 追加）
例行协议通知（reset 前后/round 边界/心跳）写 `.dispatch/runs/default/status-<agent>.log`（带时间戳追加），
不 dispatch send。仅阻断/P0-P1 发现/需决策/最终摘要才 send —— daemon 对忙碌 pane 会重复投递，
高频 send 会刷爆 root。
