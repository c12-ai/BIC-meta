## 一句话主张

**入口唯一、格式唯一、台账唯一；大脑多个、可随时重启重建。** 所有状态落文本（issues/PRD/spec/脚本），任何会话都是无状态工作者——好的 brief + 一手证据可让新会话冷启动接管任何链路（本周多次实证，含 root 会话自身的压缩重启）。

## 拓扑

```
所有人 ──→ 【总前台】（薄层：飞书群/值班入口）
              ① 按模板立 issue（现象/证据/裁定请求）
              ② 归入某条 stream（或建议开新 stream）
              ③ 回追踪链接
       ──→ 【stream S1】×2~4（厚层：全链托管者）
              issue 从立案到关闭不换手；跨 repo 派发 fix 会话；
              部署、复测闭环、飞书摘要
       ──→ 【台账 = 全局真相源】
              GitHub issues + PRD/spec + ops 脚本 + PR 依赖声明
              全局可见性在台账，不在任何会话的上下文里
```

- **按"问题流"分工，不按模块**：issue 是完整链路（本周实证：#263 跨 portal→BE→minio→Mind 四面、#277 跨 LLM 行为→BE 合并→FE 载荷），链路换手 = 传统跨组 ping-pong 复辟。
- **module owner ≠ flow owner**：owner 只在两个点介入——PR review（CODEOWNERS 质量门）与设计裁定（rule 5 冲突仲裁）。样本：#206→item-card 反转全程零会话沟通，靠 PR 评论 + PRD 裁定完成一次完整设计翻转，损失有界（一个报废 PR）。

## 核心原则（六柱，单人版实战验证）

1. **Issue 是协作总线**：现象→根因→PR→部署 sha→复测点全程可溯；人和 agent 共读写。
2. **author≠verifier**：修复者 FACTS/JUDGMENT 分离收尾；验收权在提出人；合并靠内容核验（squash 正文 issue 号 + 具名测试在 main），不靠自述。
3. **裁定即入库**：owner 拍板当天进 PRD/issue；agent 间靠文档对齐，不靠转述。
4. **确定性归脚本，判断留给人**：`make update`（台架）/`ops/field/update.sh`（现场）/CI 守卫负责执行与拦停；LLM 只在拦停点介入。
5. **并行有纪律**：worktree 隔离；同文件互 ack 圈地；PR 声明 `Depends on:`（update.sh 已消费）；高交集任务串行。
6. **事故飞轮**：每次翻车 24h 内固化为代码级守卫（本周样本：活体检出劫持→worktree 铁律、done 丢失→单行摘要纪律、localhost 烘焙→CI 守卫、MQ 残留→init-data 自愈、lab-404 拒启→reconciler 吸收）。

## 跨流沟通三层机制（全文本，无会话依赖）

1. **事前——结构性变更先宣告**：动工前落宣告工件（带模块标签的 issue / Trellis 任务 + PRD 裁定）。每 repo 一个置顶「热区看板」issue（谁、什么面、预计窗口）。**S1 派发检查单强制项：写 brief 前查热区看板与开放宣告。**
2. **事中——在途可见**：早开 draft PR（`gh pr list` 即雷达）；`Depends on:` 声明；spec 即广播（rule 10：契约改动同变更集更新 spec）。
3. **事后——仲裁**：rule 5 挑明给模块裁定 owner，选一不掺和。最坏代价有界 = 报废一个 PR；用偶发报废换掉常态跨流会议，agent 时代重做便宜、协调不便宜。
4. **残余风险（诚实声明）**：同日竞速不可根除（fe242 案：派发早于裁定落库数小时）。缓解 = 宣告先行 + 派发时刻查板 + 每日 sweep。

## 模板三件套（正式版另附文件，此处为骨架）

- **issue 模板**：现象（原话+截图）/ 一手证据（file:line、DB、日志）/ 定性方向（分叉假设，不预设结论）/ 修复要求 / 二元验收 / Refs。
- **brief 模板**：任务书指针 / 仓+worktree+分支 / 调查先行节 / 修复口径（已裁定勿重议项显式标注）/ 二元验收（含具名测试与内容核验）/ 并行知会（同仓在跑+互 ack 要求）/ 收尾纪律（done 单行短摘要+核退出码；长版落 issue 评论）。
- **收尾模板**：FACTS（可核验：sha/计数/证据链）与 JUDGMENT（自认可能有误）严格分离；未做之事显式列出（fail-loud）。

## S3→S1 入梯

新成员先「人肉跑 S3」：领一份 brief 用自己的 Claude 修一个 issue，体验全套纪律（worktree/具名测试/内容核验/收尾格式）；熟练后升 S1 托管一条 stream。比「给你个模块自己看」的传统入职快且可度量。

## rev2 增补（2026-07-13 讨论：中央运行时 / 接入层身份 / 阶段模型 / 双生命周期）

### 阶段模型（上位框架，先于状态机）

**开发与修复的区别全在前半，后半共享一条管线**（实证：item-card 功能 PR 与当日修复 PR 走完全相同的实现→合并→部署→验证门）：

```
阶段 A 进入        阶段 B 设计/裁定       阶段 C 共享管线           阶段 D 沉淀
fix：观察到缺陷 ──→ 轻/常无（分叉裁定）─┐
dev：产品意图  ──→ 重（PRD/交互/契约）─┼→ 实现→合并→部署→验证 ──→ 守卫/spec/教训
ops：环境变更  ──→ 中 + 不可逆性审批门 ─┘
```

开发者各阶段的不可替代供给：A=观察/意图；B=**裁定**（产品品味、域真相、风险接受度）；C=review 与验收；D=基本缺席（agent 固化）。角色变化的一句话：从"写代码的人"变成"提供域真相与裁定的人 + 验收的人"。

绿地 vs 褐地：本流程为褐地形（一手证据/回归护栏/内容核验均预设既存行为）。绿地骨架不变、阶段 B 加重（spec/契约先行），方法论细化列为开放问题 #13。

### 中央运行时主机（开放问题 #6 的解：集中而非分发）

- 一台服务器承载全部 claude/codex 会话（dispatch daemon + tmux + 看门狗 + checkouts 平移即可，orin 级硬件足够）；
- **飞书桥 = 总前台物理形态**：bot 收消息 → 按 issue/线程路由 → 注入/唤起对应会话（既有 send-keys 机制的自动化）；
- 配套共享测试环境：运行时主机旁一套 compose 全栈（资产 = ops/field/ 全套 + 容器化 mock），退役"开发者笔记本即台架"的单点；
- 看板零自建：GitHub Projects 挂 issues，列=生命周期状态，label 自动流转，按人过滤即个人视图；
- 故障域：主机挂=全体 agent 停摆，可接受——台账即真相，恢复=会话冷启动重建（已验证性质）。

### 接入层身份（开放问题 #3 的解向）

- 会话层只见任务；issue 的 author/Reported-by 即 issue→user link（GitHub 原生）；
- 追问确认：会话在 issue 评论 @对应人 + 飞书桥推 DM；
- 接入控制：飞书群成员资格 + feishu↔github 映射表（桥配置）；
- 边界：**裁定必须来自有权者**——初期社会约束（裁定公开在 issue），桥留 surface→ruling owner 表作升级路径；
- 运行时用机器人 GitHub 账号：agent 产出与人类产出天然分离，审计免费。

### Issue 生命周期状态机（阶段 C 的展开 + fix 形阶段 A/B）

| # | 状态 | 谁 | 动作与产物 | 出口 |
|---|---|---|---|---|
| 1 | 报告 | 任何人 | 前台按模板立 issue，链接 reporter | 现象+证据在案 |
| 2 | 归流 | 值班 S1 | stream 标签、优先级、查热区看板、并重复案 | 托管 S1 确定 |
| 3 | 调查 | S1 派发会话 | 一手证据、分叉判定落评论 | 根因明 或 升裁定 |
| 4 | 裁定(按需) | ruling owner | 选项+推荐 → owner 拍板；契约级同步 PRD | 裁定入库 |
| 5 | 实现 | fix 会话 | worktree、具名测试、Depends on 声明 | PR CI 绿 |
| 6 | 合并 | surface owner | CODEOWNERS 门 + 内容核验 | squash 入 main |
| 7 | 部署 | 脚本+部署锁 | 测试环境自动 / 现场合批 | 部署 sha 回填 |
| 8 | 复测 | reporter | 通过关闭 / 带证据打回→3 | 关闭 或 打回 |
| 9 | 固化(事故类) | S1 | 守卫/脚本/文档 PR + 教训入库 | 守卫合并 |

超时规则：复测积压>N天→集中复测日或标记"已部署未验证"；裁定等待>N天→前台升级。

### 会话生命周期与铁律

spawn（凭 brief 冷启动）→ 工作 → 收尾（单行 done + 长版落 issue）→ 归档销毁。
铁律：**会话内不留孤知**——值得留的结论必须在 done 前落 issue/文档，否则视为没发生（"运行时可任意重启"的前提）。

## 开放问题（待逐条深入讨论，落定前本文档不可开工）

1. **总前台形态与 SLA**：飞书群值班（stream S1 兼任）vs 专职 triage bot vs 纯 GitHub 入口？非工作时间报告怎么办？
2. **stream 初版划分**：几条、边界定义、各由谁持有？跨流边界模糊的 issue（如"性能"）归属规则？
3. **权限模型**：admin merge 收回后的应急通道；agent 用谁的凭据跑（个人 PAT vs 机器账号）；现场 ssh/DB 权限的最小化分配；权限分类器（auto-mode 拦截）在多人语境下的定位。
4. **部署锁**：台架（每人自己的？共享的？）与现场的锁粒度、持锁超时、抢占规则；是否升级为 CI 触发的自动部署（main 合并即滚台架）。
5. **热区看板载体与强制力**：置顶 issue vs label vs repo 文件？宣告缺失导致报废时的归责口径。
6. **dispatch 工具链可移植性**：当前 dispatch CLI + tmux 看门狗绑定单机单人——同事如何获得同款能力？打包/文档化/或降级为「手动 spawn + GitHub 台账」的最小可用版？
7. **台账拓扑**：issue 统一落 meta（现状）vs 各 repo 分治 + meta 只留跨仓案？编号引用规范。
8. **成本与配额**：Opus 会话成本核算；个人账号 vs 团队账号；每 stream 的会话预算与超限策略。
9. **复测/验收瓶颈**：当前 35 条待复测积压说明验收是最窄处——批量复测机制（复测批次日？E2E 自动化收编 bic-e2e-runner？）与「超时未复测自动降级」规则。
10. **度量**：判断这套在团队里 work 的指标——issue lead time（报告→部署）、报废率、复测积压深度、跨流冲突次数；采集方式（台账可derive，无需额外埋点）。
11. **S1 操作员画像与培训**：全链上下文能力的最低门槛；值班轮换是否可行（S1 的 stream 记忆靠台账可交接吗——需要一次真实交接实验验证）。
12. **与 Trellis 的关系**：Drake 的任务体系（Trellis tasks）与本流程的 issue 体系如何互指不重复（现状：PRD 裁定互链，够不够？）。
